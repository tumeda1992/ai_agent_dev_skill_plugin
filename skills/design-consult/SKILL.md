# design-consult スキル

ユーザーが持ち込んだ設計上の悩みを Opus モデルのサブエージェントに渡し、
**総論だけでなく思考過程・各論・トレードオフを含む**深い分析を返す。

## 引数の処理

`$ARGUMENTS` に以下を含めてもらう（自由形式でよい）:
- 背景・コンテキスト（どんなシステムか、何を作っているか）
- 悩みの内容（どこで迷っているか）
- 選択肢（A案・B案 など。なければ「選択肢不明」でよい）

引数が薄い場合は**補足を求めず**、与えられた情報だけで Opus に渡す。
ユーザーが「もう少し情報を加えたい」と言ったときだけ追加情報を受け取る。

## Opus に渡すプロンプトの組み立て

以下のテンプレートで Opus に渡す。`$ARGUMENTS` の内容を `{consultation}` に埋め込む。

```
あなたはソフトウェアアーキテクチャと型設計の専門家です。
以下の設計上の悩みに対して、深く分析してください。

---
{consultation}
---

## 回答の要件

**総論だけで終わらせない**。以下の構成で回答すること:

1. **問いの再定義**
   - 表面の悩みの背後にある、本質的な設計上の問いを1〜2文で言語化する
   - 「なぜこれが難しいのか」の構造を明示する

2. **各選択肢の分析**
   - 各案について: メリット・デメリット・前提としている設計思想を書く
   - 「これを選ぶと将来どうなるか」の具体的なシナリオを1つ添える

3. **見落としがちな観点**
   - 提示された選択肢の外にある視点、あるいは選択肢に含まれる隠れた前提を指摘する

4. **推奨**
   - どれを選ぶか、その理由を各論まで書く
   - 「〜だからA案」ではなく「〜という条件下では A案、〜という条件下では B案」の形で条件を明示する

5. **推奨の弱点**
   - 推奨案が崩れる条件・前提の変化を1つ挙げる

回答は日本語で。長くなってよい。省略しない。
```

## 実行

上記プロンプトで `Agent` ツールを呼び出す:
- `subagent_type`: `general-purpose`
- `model`: `opus`
- `description`: 設計相談: {悩みの一言要約}
- `prompt`: 上記テンプレートに `$ARGUMENTS` を埋め込んだもの

## 結果の返し方

Opus の回答をそのままユーザーに見せる。要約・圧縮しない。
回答の末尾に一行添える:「この分析を discussion.md に記録しますか？」
