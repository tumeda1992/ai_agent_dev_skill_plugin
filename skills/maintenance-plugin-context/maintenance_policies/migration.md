# Migration / 追随ポリシー

このpluginのskillを、参考元（移植元 upstream）や参照先（このpluginを利用する具体リポジトリ）と行き来させる時の規約。
skillの新規移植・upstreamからの追随・consumerリポジトリでの改善の逆輸入など、「特定リポジトリ ⇄ このpluginのskill」の更新すべてに適用する。

## リポジトリ固有情報の除去（移植・追随時 MUST）

参考元・参照先のリポジトリから得た内容をこのpluginへ取り込む時は、**そのリポジトリ固有の情報を抜いて、汎用知識だけを記載する**。

### なぜ

- このpluginは**公開配布物**で、参考元・参照先は**非公開リポジトリのことがある**。skill本体・`.steering/` 配下の成果物・docsはこのpluginにcommitされ公開されるため、非公開リポジトリの内部情報がそのまま漏れる。
- 汎用pluginに特定リポジトリの固有名が混ざると、他リポジトリでの可搬性が落ち、読み手（未来の自分・別セッションのAI・plugin利用者）のリーディングコストが上がる。
- 「公開物に入る成果物には非公開ソースの固有情報を残さない」——移植系タスクは「参照元=固有／移植先=汎用・公開」という非対称を常に持つ。

### 抜くもの（リポジトリ固有）

- 社名・リポジトリ名・プロダクト名・チーム内コードネーム
- 絶対パス（`/Users/...` 等）、そのリポジトリ限定のディレクトリ構造・ファイルパス
- 固有のドメインモデル名・クラス名・社内略称（頭字語）
- 固有の外部サービス名・特定環境のコマンド／実行基盤（例: 特定の compose 構成、社内CIコマンド）
- 参照元リポジトリの commit ハッシュ・PR番号・issue番号

### 残すもの（汎用知識 = 取り込みの本体）

- リポジトリに依存しない原則・手順・チェック観点・失敗パターン
- 移植する**最終文言そのもの**（既に汎用化された、どのリポジトリでも成立する記述）

### やり方

- 参照元は総称で書く（例:「移植元（upstream）」「利用先リポジトリ」）。固有名で呼ばない。
- 固有例は汎用の仮名・抽象説明に置換する（例: 固有モデル名 → `UserProfile` / `PaymentMethod` のような一般的仮名、固有セキュリティ文脈語 → 「変更の起点」等の一般表現）。
- 絶対パスは相対パス・構造記述に、commit ハッシュは「成長 commit（N 回）」等の記述に置き換える。
- リポジトリ固有factが取り込みの前提として必要な場合は、pluginへ直接書かず `maintenance-plugin-context` 経由で利用先リポジトリのcontext instanceから解決する（このskill本体の役割）。

### 適用範囲

- skillファイル本体（`SKILL.md`・templates・scripts など）
- 移植作業中に作る `.steering/.../` 成果物（design.md / discussion.md / investigation.md / tasklist.md / summary.md）と、ディレクトリ名（slug）
- 取り込みに伴い作成・更新する docs

### セルフチェック

commit前に、取り込んだ範囲を `grep` で確認する。参照元の社名・リポジトリ名・絶対パス・固有モデル名・commit ハッシュがヒットしないこと。ヒットしたら汎用表現へ置換してからcommitする。

## 移植の進め方（プロセス MUST）

### 差分分析を委譲する時は方向の不変条件を渡し、結果を再検証する

移植元と移植先の差分分析をサブエージェント等へ委譲する時は、**「移植方向は 移植元 → 移植先 のみ」という不変条件を明示して渡す**。委譲結果を鵜呑みにせず、各候補を `grep` で「移植元に存在し移植先に欠落している」ことを再検証してから採用する。

- なぜ: 2ファイルを突き合わせる分析は、どちらが元でどちらが先かの前提が揺らぐと差分の帰属方向を逆に解釈しやすい（「移植先に既にある内容」を「移植元の成長」と誤報する）。
- 不変条件を使えば「移植先に存在し移植元に無い」候補は定義上ありえないと機械的に棄却できる。

### 移植する変更が移植元固有の前提に結合していないか確認する

取り込む変更が、移植元**固有の他成果物の構造・命名・運用**に暗黙結合していないか確認する。結合している場合は、そのまま移植すると移植先で機能しない。移植先の構造に **adapt**（対応する要素へ読み替え）してから取り込む。

- 例: あるロジックが「移植元の design.md の特定セクション名」を前提に動く場合、移植先の design.md 構造に合わせてセクション名・抽出元を読み替える。
- 「文言をコピーすれば動く」と考えず、その変更が依存している前提を洗い出してから移植する。
