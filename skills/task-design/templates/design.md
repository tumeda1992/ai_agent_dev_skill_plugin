## 3. 完成後の姿

<!-- この設計文書の核心。
     4つのサブセクションは「完成した世界を4つの視点で切り取ったもの」。
     どれか1つが欠けると認識が揃わない:
     - 3-1: 動的な視点（操作フロー）          → mutation の引数・呼び出し順序の認識合わせ
     - 3-2: 静的な視点（データモデル）        → データ整合性の確認
     - 3-3: 全体レイヤ（命名・モジュール境界） → public 名と配置で処理の流れが読める状態
     - 3-4: ファイル系成果物（docs・設定）    → 機能じゃない deliverable の中身
-->

### 3-1. 操作フロー

<!-- 動的な視点: 「ユーザーが何かしたとき何が起きるか」。
     フロントが何を何回呼ぶか、バックエンドで何が起きるか、データがどう変わるか。
     これがないと mutation の引数・呼び出し順序で認識がずれる。
     典型ケースを1〜3つ、番号付きステップで示す

     ケース選択の MUST:
     - 作成ケースだけでなく、削除・更新・再取得ケースも含める
       （FK を追加したら削除時の挙動、mutation を追加したら影響する画面の再取得など）
     - 操作フローは「ユーザーが何回タップ/選択するか」レベルで記述する（「画面に表示される」止まりにしない）
     - フロントのバリデーション設定はバックエンドの nullable/required 定義と照合して記載する -->

**ケース: {ユーザー操作の名前}**
```
① ユーザーが ...
② フロントが {mutation/関数}({arg}: {value}) を呼ぶ
③ サーバーで {Command/関数}.call(...) → {データ変化}
④ {次の処理}
⑤ 画面に何が表示されるか
```

### 3-2. データモデル

<!-- 静的な視点: 「完成後のデータはどうなっているか」。
     スキーマ定義（カラム名の列挙）だけでは整合性が確認できない。
     具体的な行データで「このケースは表現できるか」を確認する。
     典型ケース（正常系・未設定状態・既存の別ケース）を複数示す -->

**{テーブル/コレクション名}**
| id | {col1} | {col2} | ... |
|----|--------|--------|-----|
| 1  | ...    | ...    | ... |

**典型ケース:**
- {ケース1（完成状態）}: ...
- {ケース2（未完了・中間状態）}: ...
- {ケース3（関連なしの既存パターン）}: ...

### 3-3. 命名・公開API・モジュール境界（全体レイヤ）

<!-- 全体レイヤ: 「public 名と配置だけで処理の流れが読める状態」を作る。
     命名と配置は実装中に決めない。設計時に確定する。
     (A) 命名・公開API、(B) モジュール境界・ディレクトリ構成 の両方が揃って初めて成立。 -->

#### (A) 命名・公開API

<!-- public メソッド名・クラス名・mutation 名を、根拠付きで列挙する。
     `set_xx` / `get_xx` / `do_xx` / `handle_xx` で済まされていないかチェック。
     ドメインが何をしているかを表す動詞・名詞を選ぶ。 -->

**新設クラス・関数:**
- `{ClassName}`: {役割}
  - `{method_name}(args)`: {何をするか}
    - 根拠: {なぜこの名前か。代替案を退けた理由}

**mutation / 公開 API:**
- `{mutationName}({arg}: {Type}, {optional_arg}?: {Type})`: {説明}

**配置:**
```
{path/to/module}/   ← {責務の宣言}

```

**境界のルール:**
