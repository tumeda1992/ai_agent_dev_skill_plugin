# タスクリスト

## 🚨 タスク完全完了の原則

**このファイルの全タスクが完了するまで作業を継続すること**

### 必須ルール
- **全てのタスクを`[x]`にすること**
- 「時間の都合により別タスクとして実施予定」は禁止
- 「実装が複雑すぎるため後回し」は禁止
- 未完了タスク（`[ ]`）を残したまま作業を終了しない

### 実装可能なタスクのみを計画
- 計画段階で「実装可能なタスク」のみをリストアップ
- 「将来やるかもしれないタスク」は含めない
- 「検討中のタスク」は含めない

### タスクスキップが許可される唯一のケース
以下の技術的理由に該当する場合のみスキップ可能:
- 実装方針の変更により、機能自体が不要になった
- アーキテクチャ変更により、別の実装方法に置き換わった
- 依存関係の変更により、タスクが実行不可能になった

スキップ時は必ず理由を明記:
```markdown
- [x] ~~タスク名~~（実装方針変更により不要: 具体的な技術的理由）
```

### タスクが大きすぎる場合
- タスクを小さなサブタスクに分割
- 分割したサブタスクをこのファイルに追加
- サブタスクを1つずつ完了させる

### tasklistの更新タイミング（必須）
- **フェーズが完了したら即座に `[x]` に更新する**
- 最後にまとめて更新することは禁止
- 各タスク・サブタスクが終わったら、そのタイミングで更新する

## フェーズ3: 品質チェックと修正

### DoD（完了条件）
- 全テストがグリーン
- リントエラーがない（プロジェクト全体）
- （UI変更がある場合）最終スクリーンショットで見た目を目視確認済み

> ⚠️ スクリーンショット確認は「最後にまとめてやる」ものではない。
> UIに変更を加えた各フェーズで、そのフェーズのDoDにスクリーンショット確認を含めること。
> このフェーズでは全体の最終確認のみ行う。

### タスク

- [ ] 全テスト実行
    - [ ] すべてグリーン確認

- [ ] リント実行（新規ファイル）
    - [ ] 新規作成したファイルに対してリント実行
    - [ ] エラーがあれば修正して再実行
    - [ ] エラーゼロ確認

- [ ] リント実行（repository contextが全体lint commandを返した場合だけ）
    - [ ] 返された全体lint commandを実行
    - [ ] 新規コードが既存コードに与えた影響を確認
    - [ ] エラーがあれば修正して再実行
    - [ ] エラーゼロ確認

- [ ] （UI変更がある場合のみ）最終スクリーンショットで見た目を目視確認
    - [ ] pluginの`visual-inspector` skillをchildとして使い、スクリーンショットを撮る
    - ⚠️ `npx playwright` やPlaywright ツールの直接呼び出しは禁止。必ずpluginの`visual-inspector` skillを使うこと
    - [ ] 全体のデザイン・レイアウトが意図通りか確認
    - [ ] 問題があれば修正して再確認

## フェーズ4: ドキュメント更新

- [ ] doc-enricher スキルを利用したREADME.md を更新（必要な場合。不要な場合に実行は禁止）
- [ ] 実装後の振り返り
    - 実装・レビューを通じて生じた FB・実装のズレを discussion.md に論点として記録する
    - 各論点に対して3問で再発防止先を特定する：
      1. 「この FB・ズレが起きた根本原因——共有されていなかった知識の前提——は何か？」
      2. 「その知識はコードを読めば分かるか？読んでも分からない設計意図か？」
      3. 「どこに書けば次回この議論が不要になるか？」→ 書く（変更は合意後）

## 完了後のアクション

> ⚠️ 動作確認フェーズが完了するまでコミットを促すことは禁止。急かすことも禁止。

- [ ] （親ロードマップがある場合のみ）親の `roadmap.md` の対応フェーズを完了に更新する
    - 全フェーズが完了した場合は親 `roadmap.md` 自体のステータスも完了にする
    - ※ `summary.md` への手動更新は不要。summary.md は翌月初の自動まとめで tasklist.md のチェック状況からステータスが導出される（詳細は steering SKILL.md Step 1）

- [ ] コミット（フェーズ単位で分割）
    - MUST: まとめて1コミットにしない
    - フェーズごとに別コミット
    - 同一フェーズ内でも意味的に異なる変更は分割する
        - 例: 「DB マイグレーション」「ドメインモデル」「GraphQL」「フロントエンド」は別コミット
    - ユーザーが一部のみ OK の場合は OK の範囲だけコミットして残りは待つ
    - ユーザーが「不要」「後で」と回答した場合は以下の形で完了扱いにする:
        `- [x] ~~コミット~~（ユーザーが不要と回答）`

- [ ] GitHub contextがある時だけpushしてPRを作成する
    - コミットが1件もない場合はこのタスクを実行しない。急かすことも禁止
    - `git push -u origin {ブランチ名}`
    - steering skill directoryの`scripts/github/create_or_get_pr.sh`
