# Doc enricher

## 目的

次回の読解、探索、設計議論でコード全読みに戻らないため、コードだけでは分からない設計意図・責務・不変条件・安全な変更の型を残す。観測ログ、数値のスナップショット、行単位の説明を残すskillではない。

repository固有のinstruction、architecture文書、文書配置制約が必要な時だけ、`maintenance-plugin-context`へconsumer=`doc-enricher`、必要理由、必要section=`共通/プロジェクト指示`と`共通/アーキテクチャ文書`、確認元候補を渡す。返された範囲以外を探索・編集しない。

## 固定の安全境界

- Phase 1は提案のみ。ファイルを作成・編集しない。
- Phase 2は利用者が承認した候補だけを最小変更する。
- 新しいdocs directoryを作らない。置き場がなければ候補をDROPする。
- 既存READMEまたは既存の上位docsだけを対象にする。
- 日付、現在値、ログ、実装トリビア、すぐ腐る調査メモは残さない。

## 入力と確認

対象directory、読んだ/変えたファイル、設計議論またはレビュー指摘を受け取る。候補を出す前に、対象の既存README/docsを実際に読む。repositoryに議論記録があり、許可された範囲で読めるなら、コードからは分からない設計意図の一次情報として優先する。

## Phase 1: 提案

### 1. 素材を選ぶ

次だけを候補にする。

- モジュールの責務と非責務
- 読む入口、安定した検索語、読解順
- 不変条件・境界・依存方向
- 変更時に一緒に確認すべき関係
- なぜその構造なのかという設計軸

コード一ファイルで分かることは候補にしない。まず「この知識がなければ、次回どの判断で迷うか」を書く。

### 2. 抽象化する

候補ごとに、より上位の責務や原則へ引き上げられるかを繰り返し確認する。

- 既存文書に同じ原則があるならDROPする。
- まだない原則に到達したなら、最も狭く十分な既存README/docsへ置く。
- 一般的すぎるなら、その知識が欠けて誤った提案をした実例がある時だけ、近い上位文書の「失敗例」として残す。なければDROPする。

原則には、具体例、ありがちな失敗、その帰結を添える。固有のクラス名や関数名を外しても成立しない候補は、原則ではなく作業メモなのでDROPする。

### 3. ゲート

次を全て満たす候補だけ残す。

| gate | 確認 |
| --- | --- |
| 永続性 | おおむね半年後も正しいか |
| レバレッジ | 次回の読解・レビュー・設計の迷いを大きく減らすか |
| 非自明性 | 単一ファイルを読めば分かる内容ではないか |
| 探索性 | 読む入口または安定検索語になるか |
| 低保守 | 頻繁な書換えを要求しないか |
| 抽象度 | 特定taskだけでなく、置く文書の責務に合うか |
| 非重複 | 読んだ既存文書と重複しないか |

一つでも落ちた候補はDROPし、理由を一行で記録する。

### 4. 提案形式

候補ごとに、対象path、更新section、追記案、根拠、gate結果を短く示す。利用者に、候補ごとまたは全体の承認を求める。提案を承認とみなさない。

## Phase 2: 承認済み候補の適用

承認済みの文章だけを既存README/docsへ追記または最小修正する。物理的に近い文書ではなく、知識の抽象度に合う文書を選ぶ。変更後に重複、腐りやすい記述、既存文脈との矛盾を読み直す。

## 出力の文体

短い箇条書きを基本にし、コードで安定している用語を使う。MUSTは破ると事故る契約、SHOULDは強い推奨、MAYは任意の導線に限る。説明は「なぜ」「何をしてはいけないか」「どう判断するか」まで含める。
