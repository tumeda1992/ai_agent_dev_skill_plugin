# skill: doc-enricher（ディレクトリREADME知識の強化）

## repository固有文脈

提案前にプロジェクト指示または上位アーキテクチャ文書が必要な時は、`maintenance-plugin-context`へconsumer=`doc-enricher`、必要理由、必要fact、確認元候補を渡す。返された範囲だけを読み、固定path・固定文書名を推測しない。

## 目的（Goal）
コードリーディングやタスク遂行の後に、次回以降の読み解き／探索／レビューを楽にするための
**「永続性が高く、効果が大きいナレッジ」だけ**をドキュメント化する。

特に以下の2つの問題を防ぐことを重視する：
- **コード全読みによるコンテキスト消費**: README に設計意図が書いてあれば、コードを全部読まずに設計の前提を把握できる
- **認識ズレによる設計議論の迷走**: 「なぜこの構造か」「何の次元でバリエーションが分かれているか」が共有されていれば、設計議論でのコード全読みや認識の軌道修正が不要になる

主な置き場所は **ディレクトリ配下の README**。
やむを得ない場合のみ、**既に存在する上位ディレクトリの docs/** に置く（※新規に docs/ を作るのは禁止）。

---

## 運用モード（重要）
- デフォルト動作は **Phase 1: 提案のみ** とする（ファイル変更はしない）
- README/docs の作成・修正は **ユーザーが承認した後のみ** 実施する
- 当面は常に承認を要求する（出力が安定して不要になったら、別途ルール変更で緩和可能）
- 日本語で会話すること

---

## 対象外（Non-goals：絶対にしない）
- “薄いナレッジ”（当たり前・低レバレッジ・行単位の説明）を書かない
- “調査スナップショット”（数値・ログ・現状の羅列など、腐りやすいもの）を残さない
- ドキュメント置き場を勝手に増やさない（新しい docs/ を作らない）
- ユーザー合意なしに、ファイル作成／修正をしない

---

## 実行タイミング（When to run）
- タスク完了後
- レビュー対応後
- まとまったコードリーディングの後

---

## 必要な入力（Inputs required：作業場から取得するもの）
- ディレクトリ構造と、読んだ／触った関連ファイル
- （あれば）steering の `discussion.md` — 設計判断の経緯・議論が記録されていれば**最優先の素材**（コードを読んでも分からない設計意図が凝縮されている）
- （あれば）レビュー指摘と、「どのナレッジがコード上に書かれていなかったか」
- コードリーディング中に「どこ見ればいいの？」が何度も発生した箇所

---

## 出力（Output：3フェーズ）
### Phase 1: 提案（ファイル変更なし）

#### ステップ1: 既存ドキュメントの確認（必須）
- 関連するREADMEやdocsを**実際に読む**
- 「すでに書かれていること」を把握してから候補を出す
- 候補を出した後でなく、**出す前に**確認する

#### ステップ2: 候補の生成と抽象化（必須）

**起点: 「コードを読んでも分からない設計判断はどれか？」**
- やってしまいがちな失敗: 「具体的な観察」を並べて後でフィルタする → 最初から低品質な候補ばかりになり、ゲートで全落ちする
- 正しい起点: **コードを読めば分かることは素材にしない**。コードを読んでも「なぜ？」が残る判断・制約・設計軸だけを素材にする
- 最優先の素材: `discussion.md` の設計議論（コードには現れない意思決定の経緯が凝縮されている）

**各候補を生成するときに、以下を同時に自問する（生成前フィルタ）：**
- 「このタスクが存在しなかったとしても、この知識は書けるか？」（No → DROP）
- 「特定のメソッド名・クラス名を消しても、原則として成立するか？」（No → DROP）
- 「コードを1ファイル読めば分かることではないか？」（Yes → DROP。非自明でなければ意味がない）
- 「6ヶ月後でも正しい確率が高いか？」（低い → DROP）
- 「この知識がREADMEにあれば、次回コードを全読みせずに設計議論を始められるか？」（No → DROP）
- **MUST: 候補が生まれたら抽象化ラダーを登り切ること**

  「この知識はより上位の概念（他モジュール・他ドメイン・アーキテクチャ全体）にも適用できるか？」
  を繰り返し問い、適用できなくなるレベルで止まる。止まった位置が置き場所。

  ラダーの止まり方と判断:
  1. 「既存ドキュメントに同じ原則がある」→ Gate G（重複）→ DROP
     ただしこのセッション中にその原則を知らず誤った前提・提案をした場合は、
     既存ドキュメントの「やってしまいがちな失敗」に具体例として追記する
  2. 「まだ書かれていない原則に到達した」→ 書く（必ず具体例を添える）
  3. 「自明・一般的すぎて書けるドキュメントがない」→ 問う:
     「このセッション中にこの知識がなかったために誤った前提・提案をしたか？」
     Yes → 最も近い上位ドキュメントに「やってしまいがちな失敗」の具体例として添える
     No → DROP

  やってしまいがちな失敗: 最初に思いついたレベル（具体的なモジュール）で止まる
  例: 「OrderEntry::Root の変更ガイド」
    → 実は全ての注文集約の Root に適用できる → 注文ドメインのルール？
    → 実は集約 Root の生成・更新境界を定めるアーキテクチャ層の制約 → architecture guide が置き場所

これらを全て通過した候補だけを「候補リスト」に載せる。具体事例のまま残ったものはこの時点で DROP する。

#### ステップ3: 候補の洗練（各候補に対して）
- 「この書きっぷりは最善か？より抽象的に表現できないか？」
- 「同じ問いで解ける別のケースがあるか？あれば統合して1つの原則にできないか？」
- 「やってしまいがちな失敗の具体例が添えられているか？（抽象と具体の往復）」

#### ステップ4: 自己レビュー（ゲート）による絞り込み
- 洗練済みの候補に対してゲートを適用する（落とした理由を1行で記録）

#### ステップ5: 最終提案
- 追記／更新するファイルパス（README優先）
- 追加／更新するセクション（日本語テンプレに沿う）
- 追記する箇条書き（簡潔に、確定文で）

5) 誤った／腐るドキュメントになりそうで回避不能な場合のみ、最小限の質問

### Phase 2: 適用（ユーザー承認後のみ）
- 承認された内容だけ README / docs を作成・更新する
- 変更は最小（関係ない箇所の大規模リライトは禁止）
- 置き場所の決め方: **知識の抽象レベルに対応するドキュメントを選ぶ**（物理的に近いからといって狭いREADMEに書かない）
    - プロジェクト全体の設計原則 → repositoryの上位アーキテクチャドキュメント
    - ドメイン全体の構造ルール → ドメイン上位の README
    - モジュール固有の契約 → モジュール README
    - 既存の上位 docs/（※存在する場合のみ。新設禁止）

### Phase 3: 振り返り
- このスキル定義ファイルについて、変更の必要があるか確認
  - 不要であれば、修正は禁止

## 残してよいナレッジの型（Knowledge types allowed）
A) モジュールの役割（Why/What）＋明確な非責務（やらないこと）
B) 探索・読解の索引（どこから読むか／入口／grepで引っかかるキーワード）
C) 不変条件・制約（守るべき契約／破ると事故る条件）
D) 安全な変更ガイド（Xを変えるならYも見る、影響範囲の”型”）
E) 設計意図・設計軸（なぜこの構造か／何の次元でバリエーションが分かれているか）
   - 例: 「registerDocument 系 3 mutation は “文書の取得元” という次元でバリエーションが分かれている」
   - 例: 「異なる集約をまたぐオーケストレーションは application service が担う（個別集約の command に内包しない）」
   - コードを読めば動作は分かるが、「なぜこの設計か」は読んでも分からないものが対象
   - 設計議論でコードを全読みせずに認識を揃えるために書く

---

## 自己レビュー・ゲート（全て通過必須）
Gate A: 永続性が高い（6ヶ月後でも正しい確率が高い）
Gate B: レバレッジが高い（時間を大きく節約／レビュー指摘の再発を防ぐ／設計議論での認識ズレを防ぐ）
   - 「この知識が README にあれば、次回コードを全読みせずに設計議論を始められるか？」を問う
Gate C: 非自明（1ファイル読めば分かる程度の内容ではない）
Gate D: 探索性が上がる（grep／スキャンの導線になり、安定キーワードを提供する）
Gate E: 低メンテ（頻繁な更新が必要にならない）
Gate F: 適切な抽象レベル（特定のタスク・メソッド・クラスに依存していない原則になっている）
Gate G: 既存ドキュメントとの非重複（ステップ1で読んだ既存READMEに同等の内容がない）

どれか1つでも落ちたら：DROP（捨てる）し、理由を1行で記録する。

---

## 文体・書き方ルール（Style rules）
- 短い箇条書き優先。長文エッセイ禁止。
- コードに存在する安定した用語・名前を使う（grepに強くする）。
- MUST/SHOULD/MAY を意図的に使う：
    - MUST：不変条件／契約／安全要件（破ると止まる）
    - SHOULD：強い推奨（例外があるなら条件も書く）
    - MAY：任意のヒント（原則レア）
- 日付、"currently"（現状）、メトリクス、実装トリビアを避ける。
- “いま観測できたこと”より、“こうでなければならない制約”を優先する。

## READMEテンプレ（日本語・提案）
### 1. 概要
- このディレクトリ（モジュール）が担う役割（Why/What）
- 非責務（ここではやらないこと / やるべきでないこと）

### 2. クイックマップ（読む順・入口）
- 最初に見るべき入口（例：エントリポイント、主要クラス、主要関数）
- 「だいたいここを見れば分かる」読み順
- grep用の安定キーワード（コードに存在する名前を使う）

### 3. 不変条件・契約（Invariants / Contracts）
- MUST：破ると事故る・レビューで必ず止まる条件
- SHOULD：強い推奨（例外があるなら条件も書く）

### 4. 変更ガイド（安全に変える）
- よくある変更パターン（例：「Xを追加する」「Yを差し替える」）
- そのとき一緒に確認・修正すべき場所（if X then check Y）
- 影響範囲の“型”（スナップショットの列挙ではなく関係性）

### 5. 参照
- 既存ドキュメントへのリンク（新しい docs/ は作らない）
- 関連する上位READMEや設計資料への導線（存在する場合のみ）

---

## 安全指針（Safety）
- 永続性に自信がない記述は「弱める」か「捨てる」。
- 迷ったら、まず**抽象レベルを上げる**（書く場所を変えれば Gates を通過できるかもしれない。より上位の README やrepositoryの上位アーキテクチャドキュメントに置けば "汎用設計原則" として成立することがある）。抽象レベルを上げても通過できないなら DROP。
- “観測”を残す必要がある場合は、原則として残さない。どうしても必要なら：
    - 観測ではなく「契約」「制約」「関係性」に抽象化できないか再検討する
    - 抽象化できないなら、その項目は DROP する
