# Design: 移植元 skill 成長分の plugin への backport

> 用語: **移植元（upstream）** = 本 plugin の skill の派生元。移植後も独自進化を続けている本家。
> **plugin（本 repo）** = この配布用リポジトリ。
> ※ この design.md は公開 plugin repo に含まれるため、移植元の固有名・絶対パス・固有ドメイン名は残さない。移植する**最終文言**（汎用化済み）が成果物本体。

## 目的
移植元（本家・独自進化中）が移植後に得た成長のうち、汎用性があり plugin 配布版にも価値がある差分だけを、plugin repo の該当 skill に反映する。移植元固有の具体例・環境依存は汎用表現に言い換えたうえで取り込む。

## スコープ
- 対象 skill: `task-design`, `steering`, `doc-enricher`（`think-through` は成長なしのため対象外）
- 確定移植 6 項目（T1, T3, S1, S2, D1, S3）

**変更4: ファイル末尾（line 668 の後）に §9 を新設。** 移植元の §9 をベースに、§9-5 実例リファレンスの移植元固有パス参照を削除して汎用化する:

```markdown
---

## 9. 軽量モード（discussion 駆動）

通常モードは「実装前の不確実性をゼロにする」目的に特化しているが、docs / skill / 議論ベース逐次成果物のように「完成後の姿が紋切り型では書けない / 不要な」タスクには oversized。そのようなタスクには **軽量モード** を使う。

### 9-1. 適用条件
以下のいずれかに該当する場合に軽量モードを選ぶ:
- 実装コードが伴わず、ドキュメント / プロンプトが主成果物
- 着手前に「完成後の姿」を書き切れず、討議を重ねて決定が積み上がるタスク
- 議論駆動で逐次的に成果物を作っていく形が自然

具体例:
- docs（README / ガイド / 規範文書）の新設
- skill / プロンプトテンプレートの作成
- 設計判断を伴う規約類の議論

### 9-2. モード切替
- ユーザー明示指示 / Claude 自動判定（ユーザー合意必須）
- 通常モード開始後でも、議論途中で軽量モードへの切替を提案可
- 切替時は理由を `task-design-discussion.md` に記録する

### 9-3. 通常モードとの差分
| 要素 | 通常モード | 軽量モード |
|---|---|---|
| design.md「目的」 | MUST | ✅ MUST |
| design.md「完了条件」 | MUST | ✅ MUST |
| design.md「完成後の姿」 | MUST | ❌ 不要 |
| design.md「要件」 | MUST | ❌ 不要 |
| design.md「決定事項ストック」（D1, D2, ...） | なし | ✅ 主な中身 |
| tasklist.md | MUST | ❌ 後置可 |
| requirements.md | 必要時 | ❌ スキップ可 |
| task-design-discussion.md | 任意 | ✅ 主成果物 |
| doc-enricher 起動 | 起動しない | 起動しない |

### 9-4. 進行フロー
1. 軽量モード宣言（ユーザー明示 / Claude 提案 + 合意）
2. `design.md` を最小骨組みで作成
    - 目的 / 完了条件 / 決定事項 セクションのみ
    - 「完成後の姿」「要件」は書かない
3. 論点1 を `task-design-discussion.md` に起こす
4. 議論 → 決定
5. 決定事項を `design.md` の D1 として転記
6. 論点2 → 論点3 → ... 同様に繰り返し
7. 完了条件の全チェックボックス ✅ で task-design 完了

### 9-5. 軽量モードでの成果物の中身
- `design.md`: 「決定事項ストック」中心の構成（D1, D2, ... と決定を積み上げる）。「完成後の姿」「要件」の重量級セクションは持たない
- `task-design-discussion.md`: 論点1〜N の議論変遷を記録（提起の背景・議論の変遷・決定・ネクストアクション）

### 9-6. 軽量モードでも適用する原則
- §2-3 観点4「docs・設定・環境構築系 deliverable」の判断軸は通常通り適用
- §3 設計思想（WHY→WHAT→HOW / イテレーティブ / 合意土台 / 不確実性のためならコード / 対話）は通常モード同様に適用
```

**汎用化の判断:** §9-2 と §9-4 で移植元の `discussion.md` を plugin の呼称 `task-design-discussion.md` に統一。§9-5 は移植元固有パス参照を捨て「成果物の中身」の一般説明に置換。§9-6 の §2-3 観点4 / §3 参照は plugin にも同名セクションが存在するため有効。

### T3 — task-design Step 4 に「読み手セルフレビュー」＋「設計記述の主語の選び方」を追加【高】

**なぜ:** 「論点が全部解消 = 設計完了」と判定すると、design.md が「一般則」と「変更点一覧（結論）」の間の層（各変更がどの分類に該当しなぜそのパターンか）を欠き、tasklist 化で「なぜこの変更か」を辿れなくなる。task-design のゴール（実装者は手を動かすだけ）を脅かす穴。「主語は修正の入口にする」は複数ファイル・レイヤ横断の設計（plugin の想定起動条件）で可読性を大きく左右する汎用ライティング原則。

**対象ファイル:** `.claude/skills/task-design/SKILL.md`

**変更: Step 4 完了判定の箇条書き末尾（現 line 523 `- 2-2 ネガティブ定義3つ全てに該当しない` の直後、`**design.md 完成設計書チェック（2 段階）:**` の前）に追記。** 移植元の固有セキュリティ文脈の例を汎用化:

```markdown
- **読み手セルフレビュー**: 「実装者が後続フェーズ（tasklist 等）の各タスクを見たとき、その根拠（なぜそのパターン・なぜその変更か）を design.md 内で完結して辿れるか」を確認する
    - やってしまいがちな失敗: 「論点が全て解消 = 設計完了」と判定。tasklist 化フェーズで初めて「なぜこの変更か」が design.md から辿れないことが露呈する
    - 判断基準: design.md の修正パターン（一般則）と変更点一覧（結論）の間に、「個別の変更箇所が分類のどこに該当し、なぜそのパターンを選んだか」のレイヤがあるか

**設計記述の主語の選び方:**
- 修正の入口（ユーザー操作の起点・変更の起点）を主語にする
    - 例: 修正対象が API エンドポイントなら、design.md の節立てもエンドポイントを主語にする
    - 例: 修正対象が CLI コマンドなら、コマンドを主語にする
- やってしまいがちな失敗: コードベース都合の分類（モデル分類・レイヤ分類など）を主語にする → 修正の入口から外れた説明になり、読み手が「なぜこのモデルがいきなり出てくるのか」を追えなくなる
- 個別モデル・個別レイヤの事情は、それを扱う「入口」の節内で語る形にする
```

**汎用化の判断:** 移植元の `修正の入口（〈セキュリティ文脈〉の起点・ユーザー操作の起点）` を `修正の入口（ユーザー操作の起点・変更の起点）` に置換（移植元固有のセキュリティ文脈語を除去）。API エンドポイント / CLI コマンドの例は元々汎用なので温存。

**なぜ:** 「同一エンドポイント = 同一フェーズ」と機械的に判断すると、内部分岐（type 別等）で修正対象・修正パターンが違うのに 1 フェーズに集約され、DoD に複数ケースが並列に混在する。plugin の Step 7 には「複数操作の混在」「外部サービス混在」チェックはあるが、この「同一エンドポイント内の内部分岐」の盲点だけ欠けている。

**対象ファイル:** `.claude/skills/steering/SKILL.md`

**変更: Step 7 の第1チェック項目（現 `docker compose build…` サブ項目 の後、`- [ ] **各フェーズに DoD があるか**` の前）に追記。** 移植元の Rails 依存語を薄めて汎用化:

```markdown
    - **同一エンドポイント（controller#action 等）でも内部分岐で挙動が分かれる場合はフェーズ境界を検討する**: `case ... when` / 条件分岐などで挙動が変わる場合、分岐ごとに修正対象・修正パターンが異なるなら、コード経路をフェーズ境界として切る。「同一エンドポイント = 同一フェーズ」と機械的に判断しない

        - やってしまいがちな失敗: 同じエンドポイントだからと 1 フェーズに集約し、DoD に複数の分岐ケース（type 別等）を並列に並べてしまう
```

### S2 — steering 注意事項に「ドメイン固有名詞を略称で書かない」を追加【中】

**なぜ:** steering 成果物（design.md/tasklist.md/discussion.md）は実装者（未来の自分・別セッションの AI 含む）が読み返す前提。独自略称（社内でしか通じない頭字語）が混ざると意味を復元できずリーディングコストが跳ねる。どのドメインにも固有の略称文化があるため汎用性が高い。

**対象ファイル:** `.claude/skills/steering/SKILL.md`

**変更: 注意事項（現 line 25 `会話は日本語で行うこと。` の直後）に追記。** 移植元の悪い例/良い例は移植元固有ドメイン名なので、汎用の仮名に差し替える:

```markdown
- 会話は日本語で行うこと。
- **全成果物（design.md / tasklist.md / discussion.md 等）でドメイン固有名詞を略称で書かないこと**。クラス名・モデル名などはフルネームで書く。略称（社内でのみ通じる頭字語）はリーディングコストを上げる

### D1 — doc-enricher ステップ1に「確認範囲を対象ディレクトリに閉じない」原則を追加【低】

**なぜ:** 対象ディレクトリだけを見て「まだ書かれていない」と判断すると、上位・隣接文書（API 仕様・スキーマ定義等）に既に設計意図が書いてあるのを見落とし、重複提案 or 見落としが起きる。移植元では API 仕様書で実害を観測。原則は汎用。

**対象ファイル:** `.claude/skills/doc-enricher/SKILL.md`

**変更: ステップ1（現 line 63 `候補を出した後でなく、**出す前に**確認する` の後）に追記。** 移植元の固有パスを汎用化:

```markdown
- **確認範囲を対象ディレクトリに閉じない**: API 仕様・スキーマ定義・エラーコード定義など、隣接・上位のドキュメントに既に設計意図が書かれていることがある。対象ディレクトリだけを見て「書かれていない」と判断すると、重複提案または見落としが起きる

### S3 — steering summary.md「翌月自動まとめ方式」への変更【確定: option a 採用】

**なぜ tasklist.md からステータスを取るか:** plugin では実装完了の一次情報は tasklist.md のチェックボックス。design.md の「完了条件」は設計完了を指すため summary の「機能が完成したか」とは意味がずれる（論点3）。

**対象ファイル:** `.claude/skills/steering/SKILL.md` ＋ `templates/summary_entry.md`（削除）

**変更1: Step 1 の item 4 を差し替え**（現 line 168-172 の「summary_entry.md を参照して追記」を以下に置換）:
```markdown
4. **前月分 summary.md の自動まとめ**（前月ディレクトリが存在し、その月の `summary.md` が **未存在** のときのみ実行）
    - 対象月の **1 ヶ月前**（`.steering/[YYYY]/[YYYYMM-1]/`、年をまたぐなら前年12月）のディレクトリが存在するか確認する
    - 該当ディレクトリ配下に `summary.md` が **既に存在する** 場合は何もしない
    - **未存在** の場合のみ以下を実行:
        - 前月配下の各 steering ディレクトリを `ls` で列挙する
        - 各ディレクトリから概要・ステータスを抽出する:
            - **概要**: `design.md` の `## 1. TL;DR` 本文の最初の段落を抽出（無ければ `## 目的` の最初の段落。それも無ければ `{slug}（概要抽出不可、design.md 参照）`）

##### 自動まとめエントリ書式
（`# [YYYY]年[MM]月 Steering サマリー` ヘッダ + 各 steering につき）
```markdown

**ステータス:** {完了 / 未完了 / 不明}

---
```
エントリは slug（リンク）+ 概要 + ステータスのみ。種別 / 関連 / 詳細フィールドは持たない。詳細は各 steering の design.md / discussion.md を参照。
