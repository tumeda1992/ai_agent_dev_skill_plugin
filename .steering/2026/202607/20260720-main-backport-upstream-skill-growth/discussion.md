# discussion — 移植元 skill 成長分の backport

> 用語: **移植元（upstream）** = 本 plugin の skill の派生元。移植後も独自進化を続けている本家。
> **plugin（本 repo）** = 移植元から汎用化してコピーした側。
> ※ 移植元の固有名・commit・固有ドメイン名は残さない（公開 plugin repo に含まれるため）。

## 論点1: backport 候補の検証で「方向が逆」の誤報が出た（T2）

**提起の背景:** 差分分析を委譲したサブエージェントが task-design の backport 候補として「§5 Step 3『論点の粒度と質問の材料』が移植元にあって plugin に無い」と報告した。しかし移植は「移植元 → plugin」の方向なので、plugin に既にある内容を移植元から持ってくることはありえない。報告の方向が疑わしかった。

### 議論の変遷

#### 事象の記述
- サブエージェントのレポートは T2 を「移植元に存在し plugin に欠けている」候補として挙げた。
- 実際に grep で検証すると、「論点の粒度」「1つの決定」「判断材料なし」等の該当文言は **移植元側に 0 件**、**plugin 側 `task-design/SKILL.md:465-469` に存在**していた。

#### 原因の追跡
- なぜ誤報が出たか: サブエージェントが diff の追加/削除方向を取り違えた（plugin にしかない行を「移植元の成長」と読んだ）。
- なぜ取り違えが起きうるか: 2つのファイルを読んで比較する際、どちらが移植元でどちらが移植先かの前提が揺らぐと、差分の帰属方向を逆に解釈できる。
- なぜ検証で気付けたか: 「移植方向は 移植元 → plugin のみ」という不変条件があるため、「plugin にあって移植元に無い」ものは定義上 backport 対象になりえない。この不変条件で機械的にフィルタできる。

**決定:** T2 は backport 対象から取り下げ（plugin に既存）。他の候補（T1, T3, S1, S2, S3, D1）はすべて grep で「移植元に存在・plugin に欠落」を再検証済み。

**ネクストアクション:** design.md の移植項目一覧から T2 を除外する。

## 論点2: S3（summary.md 翌月まとめ方式）は plugin の design.md 構造と結合しており、素直な移植ができない

**提起の背景:** ユーザーは S3（移植元が採用した「summary.md を翌月初に自動まとめ」方式）を backport 対象に選定した。選定時点では「Step 1 のロジック差し替え + summary_entry.md 削除」程度の変更と認識していた。しかし移植元の自動まとめロジックを精査すると、design.md の特定セクションからテキストを抽出する設計になっており、plugin 側の design.md はそのセクションを持たないことが判明した。

### 議論の変遷

#### 事象の記述
- 移植元の自動まとめは、前月各 steering の design.md を開き、
  - **概要** = `## 目的` セクション本文の最初の段落
  - **ステータス** = `## 完了条件` セクションのチェックボックス判定
  を抽出して summary.md を生成する。
- plugin の design.md テンプレート（`task-design/templates/design.md`）の見出しは `## 1. TL;DR` / `## 完成後の姿` / `### 受け入れ基準` などで、**`## 目的` も `## 完了条件` も存在しない**。
- そのまま移植すると、抽出は常に失敗し「(概要抽出不可、design.md 参照)」「ステータス: 不明」になる。

#### 原因の追跡
- なぜ結合しているか: 移植元の自動まとめは移植元側の design.md（軽量モードの design.md は `## 目的`/`## 完了条件` を持つ。§9-3 差分表参照）に合わせて設計された。抽出元セクション名が design.md 構造にハードコードされている。
- なぜ plugin で壊れるか: plugin の通常モード design.md は task-design 由来の別テンプレートで、セクション名が異なる。
- さらに: 移植元は「親子 steering の関連性を summary.md に記録しない」方針だが、plugin は summary.md に roadmap/親子リンクを積極的に記録する運用（`steering/SKILL.md:253-257`）を持つ。S3 を完全移植すると、この plugin 側の機能を削ることになる。

#### 根本原因₀ + 提案₀
- **根本原因₀**: S3 は「summary.md の書き込みタイミング変更」に見えて、実際は「design.md のセクション構造」「roadmap/親子運用」と結合した3点セットの変更。
- **提案₀（現時点・未決）**: S3 の扱いをユーザーに確認する。選択肢 a) 完全移植（adapt あり） / b) 見送り / c) 中間（都度追記のまま軽量調整）。

**決定:** **a) 完全移植（adapt あり）を採用**（ユーザー選択）。翌月まとめ方式を採用し、抽出元を plugin の成果物構造に合わせて書き換える。`summary_entry.md` 削除。roadmap/親子の summary 記録は廃止（移植元に倣い roadmap.md / design.md 側に寄せる）。

**ネクストアクション:** option a の具体設計を論点3で詰める（抽出元セクションの確定が必要）。

## 論点3: option a で「概要」「ステータス」をどこから抽出するか

**提起の背景:** S3 の完全移植（論点2 で a 採用）を実装するには、自動まとめが plugin の成果物のどこから概要・ステータスを読むかを確定する必要がある。移植元は design.md の `## 目的`/`## 完了条件` を読むが、plugin にはこの2セクションが無い。plugin の成果物構造（design.md はモードで2種、実装進捗は tasklist.md）に合わせた抽出元を決めないと自動まとめが機能しない。

### 議論の変遷

#### 事象の記述
- plugin の design.md は2形態: 通常モード（`## 1. TL;DR` / `## 完成後の姿` / `### 受け入れ基準`）と軽量モード（§9: `目的` / `完了条件` / `決定事項`）。
- 実装の進捗（完了/未完了）は plugin では **tasklist.md のチェックボックス** が権威。design.md の「完了条件」は「設計の完了」であって「実装の完了」ではない。
- roadmap steering は tasklist.md を持たず roadmap.md のみ。

#### 原因の追跡
- なぜ移植元は design.md `## 完了条件` を見たか: 移植元の自動まとめは軽量モード design.md（目的/完了条件を持つ）前提で設計されたと推測。
- なぜ plugin では tasklist.md を見るべきか: plugin では「実装完了」の一次情報が tasklist.md にあり、design.md 完了条件は設計完了を指すため、summary の「ステータス」（機能が完成したか）とは意味がずれる。

#### 根本原因₀ + 提案₀
- **根本原因₀**: 抽出元は「その情報の一次ソース」を選ぶべき。概要は design.md、実装ステータスは tasklist.md。移植元のセクション名をそのまま持ち込むと二次情報・不在セクションを読むことになる。
- **提案₀（現時点）**:
  - **概要**: design.md の `## 1. TL;DR` 本文の最初の段落。無ければ `## 目的`（軽量モード）。それも無ければ `{slug}（概要抽出不可、design.md 参照）`。
  - **ステータス**: tasklist.md のチェックボックスで判定。全 `[x]` → `完了` / `[ ]` が残る → `未完了` / tasklist.md 不在（roadmap 等）→ `不明`。
  - **エントリ書式**: `## [{slug}](./{slug}/)` + `**概要:**` + `**ステータス:**` のみ（種別/関連/詳細フィールドは持たない。移植元に倣う）。
  - **roadmap/親子**: summary.md には記録しない。roadmap.md / 各 design.md を参照。

**決定:** 提案₀ で確定（design レビューでユーザー最終確認済み）。

**ネクストアクション:** design.md の S3 セクションを「確定移植（option a 詳細）」に記載。steering SKILL.md の変更箇所を列挙（Step1 item4 差し替え / summary_entry.md 削除 / Step6 roadmap-summary ロジック削除 / 完了後アクションの summary 更新タスク削除 / branch 命名は現状維持）。

## 論点4: steering 成果物が公開 plugin repo に入るため、移植元の固有表現を残さない

**提起の背景:** 本 steering の成果物（investigation.md / discussion.md / design.md / summary.md）と steering ディレクトリ名（slug）に、移植元リポジトリの固有名（社名・リポジトリ名）・絶対パス・固有ドメインモデル名・commit ハッシュを埋め込んでいた。この `.steering/` は git tracked で、公開配布用 plugin repo に commit される。移植元は非公開の社内リポジトリのため、その内部情報が公開物に漏れる。

### 議論の変遷

#### 事象の記述
- ユーザーから「移植元の汎用知識以外のリポジトリ固有の表現を slug 含めて steering ディレクトリから消してほしい」と指摘を受けた。
- 漏れていた箇所: ディレクトリ slug（移植元リポジトリ名を含んでいた）、summary.md の概要、investigation/discussion/design の本文（社名・リポジトリ名・絶対パス・固有ドメイン名・commit ハッシュ）。

#### 原因の追跡
- なぜ埋め込んだか: 調査時に移植元の実ファイルを読み、そのパス・commit・固有例をそのまま根拠として転記した。
- なぜ問題か: steering ディレクトリが公開 repo に含まれる前提を、成果物執筆時に意識していなかった。移植作業の性質上「移植元は非公開・移植先は公開」という非対称があるのに、根拠の粒度を移植元の生情報のまま残した。

#### 根本原因₀ + 提案₀
- **根本原因₀**: 「公開物に入る成果物には、非公開ソースの固有情報を残さない」という原則を、移植系タスクの成果物執筆時に適用していなかった。
- **提案₀（採用）**:
  - slug を `backport-upstream-skill-growth` に変更（ディレクトリリネーム）。
  - 移植元を「移植元（upstream）」と総称し、社名・リポジトリ名・絶対パスを除去。
  - commit ハッシュを「成長 commit（N 回）」の記述に置換。
  - 移植元固有のドメイン例（セキュリティ文脈語・固有モデル名・外部サービス名・API 仕様ディレクトリ名等）を汎用表現に置換 or 削除。
  - backport する**最終文言**（既に汎用化済み）は保持 = これが「汎用知識」であり成果物の本体。

**決定:** 提案₀ で確定・適用済み。

**ネクストアクション:** Step 9（振り返り）で「移植系タスクの成果物は公開先前提で書く」原則を steering SKILL.md に足すべきか検討する（この論点4 自体が再発防止の素材）。
